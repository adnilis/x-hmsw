# 架构设计

本文档详细介绍 x-hmsw 向量数据库的系统架构和设计理念。

## 整体架构

x-hmsw 采用分层架构设计，各层职责清晰，易于扩展和维护。

```
┌─────────────────────────────────────────────────────────┐
│                    Application Layer                     │
│                    (用户应用代码)                         │
├─────────────────────────────────────────────────────────┤
│                      API Layer                          │
│                   (QuickDB 接口)                         │
├─────────────────────────────────────────────────────────┤
│                   Interface Layer                       │
│              (VectorDB, Index 接口)                      │
├─────────────────────────────────────────────────────────┤
│                 Embedding Layer                         │
│         (TF-IDF, BM25, OpenAI Embeddings)               │
├─────────────────────────────────────────────────────────┤
│                    Index Layer                          │
│         (HNSW, IVF, Flat, ANN 索引实现)                 │
├─────────────────────────────────────────────────────────┤
│                 Compression Layer                       │
│            (PQ, SQ, Binary 压缩)                        │
├─────────────────────────────────────────────────────────┤
│                   Storage Layer                         │
│      (Badger, BBolt, Pebble, Memory, Mmap)             │
├─────────────────────────────────────────────────────────┤
│                    Utils Layer                          │
│         (SIMD, Pool, Math, Concurrency)                 │
├─────────────────────────────────────────────────────────┤
│                  Infrastructure                         │
│              (Metrics, Logging, Serialization)          │
└─────────────────────────────────────────────────────────┘
```

## 各层详解

### 1. API Layer

**职责**: 提供简化的用户接口

**核心组件**:
- `QuickDB`: 开箱即用的简化接口
- 自动保存功能
- 默认配置

**设计理念**:
- 简单易用，隐藏复杂细节
- 提供合理的默认配置
- 支持快速原型开发

**文件位置**: [api/api.go](../api/api.go)

### 2. Interface Layer

**职责**: 定义统一的接口规范

**核心接口**:
- `VectorDB`: 向量数据库核心接口
- `Index`: 索引接口
- `Storage`: 存储接口
- `Logger`: 日志接口

**设计理念**:
- 面向接口编程
- 易于扩展和替换实现
- 解耦各层依赖

**文件位置**:
- [interface/vector_db.go](../interface/vector_db.go)
- [interface/index.go](../interface/index.go)
- [interface/logger.go](../interface/logger.go)

### 3. Embedding Layer

**职责**: 提供文本向量化功能

**向量化方法**:
- **TF-IDF**: 经典的词频-逆文档频率向量化
- **BM25**: 改进的 TF-IDF，包含多种变体
  - BM25: 标准版本
  - BM25L: 长度归一化改进版本
  - BM25+: 添加 delta 参数改进
  - BM25F: 支持字段加权
- **OpenAI Embeddings**: 调用 OpenAI API 获取高质量向量

**设计理念**:
- 统一的 Vectorizer 接口
- 支持增量更新（BM25）
- 无需外部依赖（TF-IDF、BM25）
- 灵活的配置选项

**文件位置**:
- [embedding/tfidf.go](../embedding/tfidf.go)
- [embedding/bm25.go](../embedding/bm25.go)
- [embedding/bm25l.go](../embedding/bm25l.go)
- [embedding/bm25plus.go](../embedding/bm25plus.go)
- [embedding/bm25f.go](../embedding/bm25f.go)
- [embedding/openai.go](../embedding/openai.go)

## 核心设计模式

### 1. 接口隔离原则

每个模块都定义清晰的接口，依赖接口而非具体实现：

```go
// Index 接口
type Index interface {
    Insert(id int, vector []float32) error
    Search(query []float32, k int) []types.IndexSearchResult
    Delete(id int) error
    Count() int
}

// Storage 接口
type Storage interface {
    Get(id string) (*Vector, error)
    Put(vec *Vector) error
    Delete(id string) error
    // ...
}
```

### 2. 适配器模式

为不同的实现提供统一的接口适配：

```go
// HNSW 索引适配器
type HNSWIndexAdapter struct {
    graph *hnsw.HNSWGraph
}

func (a *HNSWIndexAdapter) Insert(id int, vector []float32) error {
    a.graph.Insert(id, vector)
    return nil
}
```

### 3. 工厂模式

通过工厂方法创建不同的实现：

```go
func NewIndex(indexType IndexType, config IndexConfig) (Index, error) {
    switch indexType {
    case HNSW:
        return NewHNSWIndex(config), nil
    case IVF:
        return NewIVFIndex(config), nil
    case Flat:
        return NewFlatIndex(config), nil
    default:
        return nil, fmt.Errorf("unsupported index type: %s", indexType)
    }
}
```

### 4. 对象池模式

复用临时对象，减少内存分配：

```go
type IntSlicePool struct {
    pool sync.Pool
}

func (p *IntSlicePool) Get() []int {
    return p.pool.Get().([]int)
}

func (p *IntSlicePool) Put(slice []int) {
    p.pool.Put(slice)
}
```

## 数据流

### 插入流程

```
用户调用 Insert()
    ↓
API Layer (QuickDB)
    ↓
Interface Layer (VectorDB)
    ↓
Embedding Layer (如果需要文本向量化)
    ↓
Index Layer (Index.Insert)
    ↓
Compression Layer (可选)
    ↓
Storage Layer (Storage.Put)
    ↓
持久化存储
```

### 搜索流程

```
用户调用 Search()
    ↓
API Layer (QuickDB)
    ↓
Interface Layer (VectorDB)
    ↓
Embedding Layer (如果需要文本向量化)
    ↓
Index Layer (Index.Search)
    ↓
计算距离（SIMD 加速）
    ↓
返回 Top-K 结果
    ↓
Storage Layer (可选，获取完整数据)
    ↓
返回给用户
```

## 并发控制

### 读写锁

使用读写锁保护共享数据：

```go
type HNSWGraph struct {
    mu sync.RWMutex
    // ...
}

func (h *HNSWGraph) Insert(id int, vector []float32) *Node {
    h.mu.Lock()
    defer h.mu.Unlock()
    // ...
}

func (h *HNSWGraph) Search(query []float32, k int) []SearchResult {
    h.mu.RLock()
    defer h.mu.RUnlock()
    // ...
}
```

### 细粒度锁

对热点数据使用细粒度锁：

```go
type HNSWGraph struct {
    levelLocks []sync.RWMutex
    // ...
}

func (h *HNSWGraph) searchLayer(level int, ...) {
    h.levelLocks[level].RLock()
    defer h.levelLocks[level].RUnlock()
    // ...
}
```

使用位集合替代 map，减少内存占用：

```go

    data []uint64
}

func (bs *BitSet) Set(index int) {
    word, bit := index/64, uint(index%64)
    bs.data[word] |= 1 << bit
}
```

## 总结

x-hmsw 的架构设计遵循以下原则：

1. **分层清晰**: 各层职责明确，易于理解和维护
2. **接口驱动**: 面向接口编程，易于扩展和替换
3. **高性能**: 通过多种优化手段提升性能
4. **可观测**: 完善的监控和日志系统
5. **易用性**: 提供简化的 API 接口

这种架构设计使得 x-hmsw 既保持了高性能，又具有良好的可扩展性和可维护性。
