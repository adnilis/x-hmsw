# 项目性能优化建议

基于pprof分析结果，以下是x-hmsw项目的性能优化建议。

## 1. 热点分析

### CPU热点函数
| 函数 | 占比 | 耗时 | 优先级 |
|------|------|------|--------|
| `simd.DotProduct` | 40.60% | 2.57s | 高 |
| `HNSW.searchLayer` | 7.74% | 0.49s | 高 |
| `container/heap.down` | 6.64% | 0.42s | 中 |
| `bitset.Get` (inline) | 4.74% | 0.30s | 中 |
| `container/heap.up` | 3.48% | 0.22s | 中 |

### 内存分配
- **GC次数**: 85次
- **总分配**: 1447 MB
- **GC频率**: 高
- **主要问题**: 频繁的内存分配导致GC压力大

## 3. 待优化项

### ⚡ 3.1 SIMD指令集优化

#### 问题描述
当前SSE2优化代码存在，但未充分利用现代CPU的AVX2指令集。

#### 优化方案

**方案A: 启用AVX2优化**
```go
// 在 simd/avx2.go 中使用内联汇编
// 或者使用 golang.org/x/sys/cpu 检测CPU特性

import "golang.org/x/sys/cpu"

func DotProduct(a, b []float32) float32 {
    if cpu.X86.HasAVX2 {
        return dotProductAVX2(a, b)
    }
    if cpu.X86.HasSSE2 {
        return DotProductSSE2(a, b)
    }
    return DotProductGeneric(a, b)
}
```

**预期提升**: 20-30%

#### 优先级: 高
**预计工作量**: 2-3天

### ⚡ 3.3 向量归一化优化

#### 问题描述
每次搜索都调用`normalizeVector`，产生额外的内存分配。

#### 优化方案

**方案A: 存储归一化向量**
```go
type Node struct {
    ID              int
    Vector          []float32  // 原始向量
    NormalizedVector []float32 // 归一化向量（缓存）
    Level           int
    // ...
}

// 在插入时预计算并缓存归一化向量
func (h *HNSWGraph) Insert(id int, vector []float32) *Node {
    normalized := h.normalizeVector(vector)
    node := &Node{
        ID:              id,
        Vector:          vector,
        NormalizedVector: normalized,  // 缓存
        // ...
    }
    // ...
}
```

**方案B: 直接在搜索时使用归一化向量**
- 避免重复计算
- 减少内存分配

**预期提升**: 10-15%

#### 优先级: 中
**预计工作量**: 1-2天

### ⚡ 3.4 批量操作优化

#### 问题描述
批量插入和搜索仍有优化空间。

#### 优化方案

**方案A: 批量预分配内存**
```go
func (h *HNSWGraph) BatchInsertBuffered(ids []int, vectors [][]float32) error {
    // 预分配节点
    nodes := make([]*Node, len(ids))
    for i, id := range ids {
        nodes[i] = h.createNode(id, vectors[i])
    }

    // 批量插入
    for i, node := range nodes {
        h.insertNode(node, ids[i])
    }
    return nil
}
```

**方案B: 批量排序优化**
```go
// 使用更快的排序算法
import "sort"

// 对于小数据集使用插入排序
func insertionSort(data []SearchResult) {
    // ...
}

// 混合排序策略
func hybridSort(data []SearchResult) {
    if len(data) < 32 {
        insertionSort(data)
        return
    }
    sort.Slice(data, lessFunc)
}
```

**预期提升**: 15-20%

#### 优先级: 中
**预计工作量**: 2-3天

### ⚡ 3.5 内存布局优化

#### 问题描述
向量和节点的内存布局不连续，影响缓存命中率。

#### 优化方案

**方案A: 数组结构体优化**
```go
// 使用分片存储提高缓存局部性
type HNSWGraph struct {
    vectors     [][]float32  // 连续存储所有向量
    nodeLevels  []int        // 连续存储所有节点层级
    nodeDeleted []bool       // 连续存储删除标记
    // ...
}
```

**方案B: 结构体填充优化**
```go
// 调整结构体字段顺序，减少内存填充
type Node struct {
    ID              int       // 4 bytes
    Level           int       // 4 bytes
    MaxLinks        int       // 4 bytes
    Deleted         bool      // 1 byte
    _               [3]byte   // 填充对齐
    NormalizedVector []float32 // 连续内存
    Vector          []float32
    Friends         [][]*Node
    mu              sync.RWMutex
}
```

**预期提升**: 10-20%

#### 优先级: 中
**预计工作量**: 5-7天

### ⚡ 3.6 并发优化

#### 问题描述
当前并发锁粒度较大，并发性能受限。

#### 优化方案

**方案A: 细粒度锁**
```go
// 使用分层锁
type HNSWGraph struct {
    levelLocks   []sync.RWMutex  // 每层一个锁
    nodeLocks    []sync.RWMutex  // 每个节点一个锁
    // ...
}

// 按层锁定
h.levelLocks[level].Lock()
// ...
h.levelLocks[level].Unlock()
```

**方案B: 无锁数据结构**
- 考虑使用`sync/atomic`优化热点路径
- 使用`sync.Pool`进一步减少锁竞争

**预期提升**: 30-50%（在高并发场景）

#### 优先级: 高
**预计工作量**: 5-7天
